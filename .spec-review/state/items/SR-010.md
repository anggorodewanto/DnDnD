## Finding (verbatim from review)
Sneak Attack `OncePerTurn` never enforced at runtime. `populateAttackFES` (`internal/combat/attack.go:1556`) never sets `AttackInput.UsedThisTurn`; the `EffectConditions.OncePerTurn` filter only trips when the caller threads a non-nil map. Only tests do. Every qualifying attack in a turn re-adds Sneak Attack dice.

Related observation (also tracked here per orchestrator instructions): Sneak Attack reaction-attack rider unverified (Batch 7, §3 medium) — verify reaction attacks during another creature's turn also satisfy the once-per-turn gate.

## Acceptance criteria
- `populateAttackFES` populates `AttackInput.UsedThisTurn` from the turn state (and/or per-character per-turn flag).
- Test: rogue makes Extra-Attack-style two attacks in one turn; sneak attack dice are added only on the first qualifying hit.
- Test: rogue's reaction attack on another turn correctly applies sneak attack if not used since the rogue's last turn ended.
- Tests pass; `make cover-check` green.

## Worker plan

### Where does per-turn used-effect state live today?
Nowhere. The `turns` table has `ActionUsed`, `BonusActionUsed`, `ReactionUsed`,
`AttacksRemaining`, etc., but nothing that tracks which once-per-turn
*feature effects* have been consumed. `EffectConditions.OncePerTurn` is filtered
inside `EvaluateConditions` (effect.go:236) against `ctx.UsedThisTurn`, but
`Service.populateAttackFES` never sets `AttackInput.UsedThisTurn` — only unit
tests do.

### What's "the turn" for a rogue's reaction attack on another creature's turn?
Spec: once-per-turn = "since your turn started". `cmd.Turn` in `Service.Attack`
is always the *active turn* (whoever is currently up), so during an enemy turn
the rogue's reaction-attack receives the enemy's Turn row. We therefore cannot
key `UsedThisTurn` off `cmd.Turn` — we must key it off the attacker (rogue)
identity and reset it when *the rogue's* own turn starts.

### How to thread `UsedThisTurn` into `populateAttackFES` without changing every caller?
Use an in-memory per-encounter / per-combatant tracker that lives on
`combat.Service` (mirrors the existing `summonedResources` + `ammoTracker`
patterns):

```
usedEffectsMu  sync.Mutex
usedEffects    map[uuid.UUID]map[uuid.UUID]map[string]bool
//             encounterID    combatantID    effectType
```

- `populateAttackFES` reads the tracker for `cmd.Attacker.EncounterID` /
  `cmd.Attacker.ID` and assigns the snapshot to `input.UsedThisTurn`.
- `Service.Attack` / `attackImprovised` mark fired once-per-turn effects after
  `resolveAndPersistAttack` returns (using `AttackResult.OncePerTurnEffectsFired`,
  a new field populated inside `ResolveAttack` from the on_damage_roll
  `AppliedEffects` list filtered to `OncePerTurn:true`).
- `createActiveTurn` (initiative.go) clears the tracker entry for the combatant
  whose turn just started — so once the rogue's next own turn begins the slate
  is wiped, but during enemy turns the rogue's slate persists, which is exactly
  the spec's "since your turn started" wording.
- Encounter cleanup: `EndCombat` / encounter-state resets can ignore the map
  (best-effort GC); the encounter ID never reappears.

This keeps the change local to `combat.Service` + a single new field on
`AttackResult`. No DB schema change.

### Acceptance criteria — all met
- `populateAttackFES` (`internal/combat/attack.go:1591-1596`) now assigns
  `input.UsedThisTurn = s.usedEffectsSnapshot(cmd.Attacker.EncounterID, cmd.Attacker.ID)`.
  Keyed on the *attacker*, not the active-turn combatant, which is the
  correct RAW reading.
- Extra-Attack test (`attack_fes_test.go:283-318`) asserts
  `second.DamageTotal == 8` (base 5+3, no 3d6 sneak) after a first attack
  hitting 23 — confirms once-per-turn is enforced on a same-turn second
  attack.
- Reaction-on-enemy-turn test (`attack_fes_test.go:330-368`) asserts
  `result.DamageTotal == 23` with `enemyTurn.CombatantID = target.ID` —
  confirms the slate is not falsely cleared by a different active turn.
- Reaction-after-own-turn-use test (`attack_fes_test.go:370-421`) fires
  sneak attack on the rogue's own turn (23), then asserts a follow-up
  reaction attack during the enemy's turn returns `8` (no re-apply).
- `go test ./...` → all packages PASS. `make cover-check` → "OK: coverage
  thresholds met"; combat package 92.9%.

### Concurrency — clean
- `usedEffectsSnapshot` (`service.go:341`), `markUsedEffects`
  (`service.go:358`) and `clearUsedEffectsForCombatant` (`service.go:378`)
  all `Lock`/`defer Unlock` around both reads and writes.
- Top-level map initialized in `NewService` (`service.go:332`); inner
  submaps lazily allocated under the lock in `markUsedEffects`. Read path
  tolerates nil submaps via Go's nil-map read semantics — safe.
- `createActiveTurn` (`initiative.go:679`) calls the locked clear helper —
  no direct map access. No goroutine spawned anywhere; no leak risk.

### Persistence — flagged as accepted risk
- Tracker is in-memory only. Bot restart mid-encounter would empty the map,
  allowing Sneak Attack to re-apply on the next attack in the same turn.
  Acceptable per spec's async-first design (short restarts tolerated) and
  matches the existing `summonedResources` / `ammoTracker` patterns, but
  worth a follow-up SR if production restarts prove frequent. No data-loss
  vector — just a transient over-grant of dice.

### Reaction-rider verification — semantically valid but cosmetic
- The reaction tests do not set any `IsReaction` flag, because
  `Service.Attack` has no reaction code path distinct from a normal attack —
  the rogue's reaction-attack is modeled by `cmd.Turn.CombatantID` pointing
  at the enemy. The tracker is keyed on `cmd.Attacker.ID`, which is exactly
  the surface under test. So the test setup, while not "marking the attack
  as a reaction" syntactically, *does* exercise the precise discriminator
  the fix introduces (attacker-keyed slate that survives across turn rows
  with different CombatantIDs). Acceptable.

### OncePerTurn generalization
- Grep `OncePerTurn:` across `internal/combat/*.go` (non-test) returns a
  single hit: `feature_integration.go:101` (Sneak Attack). The tracker
  fires for ANY effect whose `Conditions.OncePerTurn == true` because
  `ResolveAttack` filters `dmgResult.AppliedEffects` generically
  (`attack.go:566-569`). So Divine Strike / Improved Divine Smite / any
  future `once_per_turn` effect on `on_damage_roll` will Just Work — but
  no other consumer exists today to test against. Scope is correctly
  declared in the worker notes.
- Caveat: only `TriggerOnDamageRoll` effects are surfaced (the loop is
  inside the on_damage_roll branch). If a future once-per-turn effect
  triggers on `on_attack_roll` (e.g. some homebrew save-DC granters) it
  will not be tracked. Acceptable for current scope — no such effect
  exists.
