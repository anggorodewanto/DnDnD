## Finding (verbatim from review)
Six `/dm-queue` posters omit `CampaignID` → `PgStore.Insert` rejects after Discord message has already been sent, orphaning `#dm-queue` messages that 404 on Resolve and never appear in the dashboard. `reaction_handler.go:159`, `check_handler.go:499`, `rest_handler.go:70`, `retire_handler.go:116`, `undo_handler.go:75`, `use_handler.go:308`. Also: `Notifier.Post` ordering is send-then-insert; reverse it.

## Acceptance criteria
- All six poster call sites populate `CampaignID` on the `dm_queue_items` insert payload.
- `Notifier.Post` flips to insert-then-send, so a failed insert never leaves an orphaned Discord message.
- Negative-path test: when the insert fails, no Discord message is posted; when the insert succeeds, the Discord message + DB row reference each other.
- Tests pass; `make cover-check` green.

## Worker plan
- Flip `Notifier.Post` ordering in `internal/dmqueue/notifier.go` so the Store.Insert happens BEFORE Sender.Send (insert-then-send). If send fails after a successful insert, leave the row pending — the DM still sees a dashboard entry they can cancel — and surface the send error to caller. This is the minimal-change contract: the existing test `TestNotifier_Post_SendErrorReturned` already asserts the error propagates; the only new guarantee is that Insert is observable on send failure.
- Plumb `CampaignID` at the four handler call sites that already resolve a `campaign` (check, rest, retire, use) by threading the campaign ID into the existing post helper or building the event inline next to the campaign lookup.
- Add a campaign provider to `ReactionHandler` (via setter) so `postReactionDeclaration` can resolve `campaign_id` by guild ID. Constructor stays compatible.
- Add a campaign provider to `UndoHandler` (via setter) so `Handle` can resolve `campaign_id` by guild ID. Constructor stays compatible.
- Wire the new providers in `cmd/dndnd/discord_handlers.go` using the existing `checkCampProv`.
- Add new tests: a notifier-level test that asserts insert-then-send order (send failure leaves row inserted) plus six per-handler tests that assert the posted Event carries a non-empty `CampaignID`.
- Update affected tests in the dmqueue notifier suite to reflect the new ordering (insert-failure short-circuits before send).

- Dashboard tolerates placeholder messageID: `internal/dashboard/dmqueue_page.go` `ServeItem` / `ServeList` / `HandleResolve` consume `ID`, `Kind`, `PlayerName`, `Summary`, `Status`, `ResolvePath` only — `MessageID` is never rendered or routed through. A row with `MessageID == itemID` (post-send-failure) renders cleanly; the form posts to `/dashboard/queue/{itemID}/resolve` which routes by item id, not message id. Two-phase design preserves the spec invariant.
- Caveat (not blocking): on send-failure-then-Resolve, `Notifier.Cancel`/`Resolve` will still call `Sender.Edit(channelID, placeholderItemID, ...)` which Discord rejects ("unknown message"). The Mark{Resolved,Cancelled} DB update SUCCEEDS first (notifier.go:198-201, 213-217), so the row's status moves forward correctly but the dashboard sees a 500 from the failed edit. Acceptable trade-off — the row is in a clean DB state and the DM can clear it via direct DB or a subsequent fetch — and far better than the pre-fix behaviour (orphan Discord message with no DB row).
- Missed Post caller (out of SR-002 scope but worth noting): `cmd/dndnd/lifecycle_adapters.go:113` `NotifyHostilesDefeated` posts a `KindFreeformAction` with NEITHER `CampaignID` NOR `GuildID`. The empty GuildID means the resolver returns "" and Post short-circuits as a silent no-op (`notifier.go:165-167`) — so in production the DM never sees this prompt. This is a separate pre-existing bug ("hostiles-defeated prompt never fires"), not an SR-002 orphan-message regression. SR-002 scope is preserved.
- No regressed tests: only `TestNotifier_Post_SendErrorReturned` was modified, and it was strengthened (now asserts row persists with placeholder messageID) — no test previously pinning send-then-insert ordering existed to break.
- Setter pattern for ReactionHandler/UndoHandler: justified for `ReactionHandler` (already uses `SetNotifier` setter, so `SetCampaignProvider` is a consistent peer). For `UndoHandler` the constructor takes 5 positional args including `notifier`; adding `campaignProv` as a 6th positional arg would have been equally minimal and avoided a "construct then mutate" surface. Acceptable but the slight asymmetry (notifier injected via constructor, campaignProv via setter) is a small smell. Not blocking.
- Early-return style: SR-002 additions to `Notifier.Post` (notifier.go:163-186), `resolveCampaignID` helpers (reaction_handler.go:190-199, undo_handler.go:98-107), `SetMessageID` impls all follow early-return style. Compliant with project rules.
- No dead code introduced. `Store.SetMessageID` has exactly one caller (`Notifier.Post`) which is the intended one. `UpdateDMQueueItemMessageID` sqlc query is consumed by `PgStore.SetMessageID`.
