## Finding (verbatim from review)
WS Origin verification disabled in prod. `internal/dashboard/ws.go:119` `InsecureSkipVerify: true`. Same code path in dev + prod — CSRF-style cross-origin upgrade vector. (Also relevant: batch-14-ws-recovery-multi.md.)

## Acceptance criteria
- `ws.go` checks the WS upgrade `Origin` against an allowlist in production; only dev mode bypasses.
- Test: an upgrade request with a foreign `Origin` is rejected in prod mode and accepted in dev mode.
- Test: a same-origin upgrade succeeds in both modes.
- Tests pass; `make cover-check` green.

## Worker plan
- Add an opt-in origin policy to `dashboard.Handler` via a new
  `SetWebSocketOriginPolicy(allowedOrigins []string, insecureSkipVerify bool)`
  setter (early-return style). Two new private fields back it:
  `wsAllowedOrigins`, `wsInsecureSkipVerify`. Defaults preserve current
  behaviour (insecureSkipVerify=true) for back-compat with existing tests
  that drive `ServeWebSocket` through `httptest.NewServer` (the dial origin
  defaults to the `httptest` host but not all tests set it explicitly).
- `ServeWebSocket` now builds `AcceptOptions` from the configured policy:
  * `wsInsecureSkipVerify=true` → `AcceptOptions{InsecureSkipVerify: true}`
    (dev fallback — current behaviour).
  * else → `AcceptOptions{OriginPatterns: wsAllowedOrigins}` and rely on
    nhooyr/websocket's built-in same-host check + Origin matching to
    reject foreign origins with 403.
- nhooyr/websocket does not expose a custom `CheckOrigin` func, so this
  uses its native `OriginPatterns` (matched via `filepath.Match` against
  the request Origin host) — equivalent to the spec-review recommendation
  "check Origin against configured allowlist".
- Wire in `cmd/dndnd/main.go`: derive the allowlist host from `BASE_URL`;
  treat the deploy as prod when `BASE_URL` is set AND non-localhost
  (or, more robustly, when `COOKIE_SECURE=true`, which the existing code
  uses as the prod gate at line 428). Otherwise leave the handler in
  insecure-skip-verify dev mode.
- Red/green TDD adds three new tests in `internal/dashboard/ws_test.go`:
  * `TestWebSocketEndpoint_ProdMode_RejectsForeignOrigin` — sets
    allowlist, dials with `Origin: https://evil.com`, expects HTTP 403.
  * `TestWebSocketEndpoint_DevMode_AcceptsForeignOrigin` — leaves
    insecureSkipVerify=true, same foreign Origin, expects 101 upgrade.
  * `TestWebSocketEndpoint_SameOriginAccepted_BothModes` — drives the
    nhooyr Dial without overriding the Origin header so the request host
    matches; expects 101 upgrade in both modes.
- After tests pass, gate the policy from `cmd/dndnd/main.go` so production
  flips to the strict path and dev keeps the permissive path.

## Reviewer notes

### Acceptance criteria
- Foreign Origin in prod -> rejected: pass. `internal/dashboard/ws_test.go:365-381` (TestWebSocketEndpoint_ProdMode_RejectsForeignOrigin) asserts HTTP 403 from `nhooyr/websocket.authenticateOrigin` for `Origin: https://evil.example.com` against allow-list `["dashboard.example.com"]` + `InsecureSkipVerify=false`.
- Foreign Origin in dev -> accepted: pass. `internal/dashboard/ws_test.go:383-402` asserts the recorder is not 403 with `InsecureSkipVerify=true` (origin check skipped; upgrade fails on the fake Hijacker, which is the intended terminal state for this assertion).
- Same-origin in both modes -> accepted: pass. `internal/dashboard/ws_test.go:404-436` table-driven prod+dev, `Origin: https://dashboard.example.com` matches `r.Host = "dashboard.example.com"` per `authenticateOrigin` line 221 (`strings.EqualFold(r.Host, u.Host)`), no 403.
- `go test ./internal/dashboard/... ./cmd/dndnd/...` green (46s+0.7s). `make cover-check` green; thresholds met. One re-run flake in `TestRun_AuthProtectedRoutesRejectUnauthenticated` is unrelated (passes in isolation) and not in any file touched by SR-016.

### Implementation verification
- `OriginPatterns` is the canonical nhooyr/websocket field, not a typo. `accept.go:54` declares `OriginPatterns []string` on `AcceptOptions`; `accept.go:101` invokes `authenticateOrigin(r, opts.OriginPatterns)`; same-host short-circuit at `accept.go:221`. Match is case-insensitive `filepath.Match` (`accept.go:240-242`).
- `COOKIE_SECURE=true` is the real prod discriminator. `cmd/dndnd/main.go:427-429` already gates `secure := os.Getenv("COOKIE_SECURE") == "true"` for the session cookie; SR-016 reuses the same gate at `cmd/dndnd/main.go:914`. Not fabricated.
- Prod path is unconditional inside the gate: `cmd/dndnd/main.go:914-917` calls `dashHandler.SetWebSocketOriginPolicy(allowed, false)` whenever `COOKIE_SECURE=true`, so prod cannot silently retain `InsecureSkipVerify=true`. Helper returns nil for empty/unparsable/host-less BASE_URL (`cmd/dndnd/main.go:463-475`); nil allow-list with `false` is still strictly tighter than the old default (only same-host upgrades pass).
- Default `wsInsecureSkipVerify=true` (`internal/dashboard/handler.go:152-157`) is justified for back-compat with existing tests that drive `httptest.NewServer` without overriding the Origin (e.g. `TestWebSocketEndpoint_AcceptsConnection`, `TestWebSocketEndpoint_SubscribeToEncounter`). Acceptable; mitigated by the unconditional prod setter.

### Findings (non-blocking)
1. (low) BASE_URL port handling: `url.Parse("https://example.com:443").Host` == `"example.com:443"`, which becomes the allow-list literal. Browsers omit default ports in the Origin header, so a deploy with `BASE_URL=https://example.com:443` would fail origin checks for `Origin: https://example.com` (no port). `TestWsAllowedOriginsFromEnv` only covers `localhost:8080` (non-default port) and bare host; the default-port edge case is not tested. Same-host short-circuit at `accept.go:221` would also miss because `r.Host` would equal `example.com:443` only if the proxy preserves the explicit port. Worth a follow-up to strip default ports OR document "set BASE_URL without explicit default port". Not a regression: only affects pathological BASE_URL values.
2. (low) Dual-knob complexity vs CLAUDE.md "no over-engineering". A single allow-list (`nil/empty = permissive`) could have collapsed the API. Counter-argument: nhooyr already exposes both knobs orthogonally (`InsecureSkipVerify` skips entirely, `OriginPatterns` adds patterns), and the worker's choice mirrors that. The default `insecureSkipVerify=true` is what preserves the historical 12+ existing WS tests without touching them. Acceptable, but consider documenting in `SetWebSocketOriginPolicy` doc that passing `(nil, false)` is the strictest mode (same-host only) since that combination is currently exercised only indirectly via `wsAllowedOriginsFromEnv("")`.
3. (informational) The `hijackableRecorder` test helper (`internal/dashboard/ws_test.go:347-363`) cleverly exercises Accept past the origin gate without a real TCP conn. Good early-return style throughout `wsAllowedOriginsFromEnv` and `SetWebSocketOriginPolicy`. No nesting issues.

### Verdict basis
The acceptance tests pin the three scenarios required by the spec-review. Prod opt-in is unconditional behind `COOKIE_SECURE=true`. nhooyr/websocket's `OriginPatterns` is the correct mechanism. The two low-severity notes are deploy-config caveats, not correctness gaps.

VERDICT: approved
