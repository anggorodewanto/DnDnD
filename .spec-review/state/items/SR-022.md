## Finding (verbatim from review)
Wild Shape doesn't swap physical ability scores. HP/AC swap only; STR/DEX/CON of the beast form not applied.

## Acceptance criteria
- Wild Shape swaps STR/DEX/CON to the beast form's values (mental ability scores retained per spec).
- Test: druid Wild Shapes into a brown bear → STR/DEX/CON match the bear stat block; INT/WIS/CHA unchanged.
- Test: reverting Wild Shape restores the druid's STR/DEX/CON.
- Tests pass; `make cover-check` green.

## Worker plan

### Architecture context

Ability scores live on the **Character** row, not the **Combatant** row.
`Combatant` already carries `WildShapeOriginal pqtype.NullRawMessage` (a
JSONB snapshot) and `WildShapeCreatureRef sql.NullString` (beast id). The
existing `SnapshotCombatantState` already persists all six ability scores
(STR/DEX/CON/INT/WIS/CHA) under `ability_scores` on entry, and
`RevertWildShape` already clears the snapshot on revert. So:

- **Originals are already persisted** — `WildShapeSnapshot.AbilityScores`
  captures the druid's pre-transform scores. No new column / no migration.
- **Swap-on-entry is the missing piece** — `ApplyBeastFormToCombatant`
  overwrites HP/AC but does not surface the beast's STR/DEX/CON anywhere
  the rest of the code can read them.
- **Restore-on-revert "just works"** once we stop reading swapped scores
  from somewhere external: the snapshot is dropped on revert, so the
  effective scores fall back to `char.AbilityScores` automatically.

### Minimal change

Add a pure helper `EffectiveAbilityScores(combatant, druidScores,
beastScores)` that returns:
- `IsWildShaped == false` → druid scores unchanged.
- `IsWildShaped == true`  → merged scores: beast STR/DEX/CON +
  druid INT/WIS/CHA. (Per PHB Druid Wild Shape: "your game statistics are
  replaced by the statistics of the beast, but you retain your alignment,
  personality, and Intelligence, Wisdom, and Charisma scores".)

No schema change. No new combatant column. The existing snapshot JSON
already captures the originals, so revert is a no-op for the helper —
when `IsWildShaped` flips back to false, the helper returns the druid
scores from the character row again.

### Wiring (deliberately out of scope for SR-022)

Plumbing `EffectiveAbilityScores` into `/check` and `/save` (the C-5
follow-up the batch-07 review flagged) means adding a `CreatureLookup`
dependency to `CheckHandler` / `SaveHandler` and routing the combatant
through `parseCharacterData` / `parseSaveCharacterData`. That is its
own integration surface and acceptance criteria for SR-022 do not
require it. Keeping this SR to the helper + tests so the snapshot-based
swap is provably correct in isolation; the consumer wiring can be a
separate review item (tracked alongside C-5 in batch-07).

## Reviewer notes

SR-022's finding describes an observable gameplay bug — a druid in bear form should swing with bear STR — so a pure unused helper does not deliver value.
- Required to approve: plumb `EffectiveAbilityScores` through `/check` (skill/ability checks), `/attack` (STR/DEX feeding `AttackModifier`/`DamageModifier`), and `/save` (STR/DEX/CON saves). Needs a `CreatureLookup` (or cached beast-scores snapshot) injected into CheckHandler, AttackService.resolveAttackWeaponFull, and SaveHandler — sourced from `Combatant.WildShapeCreatureRef`.
- Round 2 (resubmitted): plumbing now lands in /check, /attack, /save per
  the round-1 ask. Production wiring in `cmd/dndnd/discord_handlers.go`
  injects `*refdata.Queries` into both handlers. Beast lookup degrades
  silently to druid scores (SR-006 pattern) so a missing creature row
  never blocks a roll. Follow-up scope deferred: routing /attack through
  the beast's own stat-block attacks (multiattack: bite + claws) is a
  deeper rework and is recorded as SR-069 in the worker summary — the
  current change already merges beast STR/DEX/CON into PC-side weapon
  swings made while Wild Shaped, which is the SR-022 acceptance bar.
