## Finding (verbatim from review)
`pending_queue_count` orphan field. `CombatManager.svelte:878,900` reads `enc.pending_queue_count` for "queued" badges but `workspaceEncounterResponse` never returns it and the field exists nowhere in `internal/`. Phase 94a done-when "badges for pending #dm-queue items" silently broken.

## Acceptance criteria
- `workspaceEncounterResponse` returns `pending_queue_count` populated from a real query against `dm_queue_items` for the encounter/campaign.
- `CombatManager.svelte` renders the badge using the populated value.
- Test (Go side): a pending queue row produces a `pending_queue_count` ≥ 1 in the response.
- Tests pass; `make cover-check` green.

## Worker plan
- Schema discovery: `dm_queue_items` rows are linked to `campaign_id` only (no
  encounter_id column). Per SR-032 prompt fallback, aggregate pending count
  by campaign — every encounter on the workspace response for the same
  campaign receives the same `pending_queue_count`. This matches the Phase
  94a UX intent (Combat Workspace shows aggregated DM-queue backlog) and
  avoids needing a join through `pending_actions.dm_queue_item_id` that
  would miss free-form DM-queue items (skill-check, etc.).
- Add a new sqlc query `CountPendingDMQueueItemsByCampaign` against
  `db/queries/dm_queue_items.sql` (`SELECT count(*) ... WHERE campaign_id = $1
  AND status = 'pending'`). Regenerate `internal/refdata` via `sqlc generate`.
- Extend `combat.WorkspaceStore` with a single method
  `CountPendingDMQueueItemsByCampaign(ctx, campaignID) (int64, error)` —
  matches sqlc's return type for `:one`/`count(*)`. Adapter in
  `cmd/dndnd/main.go` is satisfied automatically because the embedded
  `*refdata.Queries` will gain the new method post-`sqlc generate`.
- Add `PendingQueueCount int32` field with `json:"pending_queue_count"` to
  `workspaceEncounterResponse`. Populate it inside `GetWorkspace` once per
  request (single query per campaign), then broadcast the value into every
  encounter response so the existing UI badge logic (`enc.pending_queue_count`
  > 0) lights up. Cast int64 → int32 with bounded conversion to stay JSON-safe
  on tiny values; no realistic risk of overflow given queue size.
- Red/green TDD:
  - RED #1: `GetWorkspace_PopulatesPendingQueueCount` — store returns count = 2,
    every encounter response has `pending_queue_count == 2`.
  - RED #2: `GetWorkspace_PendingQueueCount_ZeroWhenNoPending` — store returns
    0, response has `pending_queue_count == 0` (resolved/cancelled rows
    naturally excluded by the `WHERE status='pending'` clause in the query).
  - RED #3: `GetWorkspace_PendingQueueCount_StoreError` — store returns
    error, handler responds 500.
  - GREEN: add struct field + store method + handler wiring; tests pass.
- Verify `go test ./...` and `make cover-check` green; do not commit.
