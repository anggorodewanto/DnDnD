## Finding (verbatim from review)
Magic items never feed combat / save / turn pipelines. All three callers of `BuildFeatureDefinitions` (`combat/attack.go:1584`, `combat/turnresources.go:262`, `discord/save_handler.go:281`) pass zero `extraDefs`. `magicitem.CollectItemFeatures` is correct and unit-tested but never invoked in production. +1 longsword adds nothing; Cloak of Protection adds nothing; Boots of Speed don't change movement. F-14 `modify_speed` parsing is correct but unreachable. Phase 88a "done when" unmet.

## Acceptance criteria
- All three call sites pass `magicitem.CollectItemFeatures(character)` (or equivalent) as `extraDefs` to `BuildFeatureDefinitions`.
- Integration tests assert: +1 longsword adds +1 to attack roll & damage; Cloak of Protection adds +1 to saves & AC; Boots of Speed double movement when activated.
- F-14 `modify_speed` from a magic item now reaches the speed-resolution path.
- Tests pass; `make cover-check` green.

## Worker plan

### Char availability at each call site
All three callers already have the full `refdata.Character` row (which carries
`Inventory pqtype.NullRawMessage` and `AttunementSlots pqtype.NullRawMessage`):
- `combat/attack.go:1584` — `populateAttackFES(..., char *refdata.Character, ...)`
- `combat/turnresources.go:262` — `turnStartSpeedBonus(char refdata.Character)`
  (the caller `ResolveTurnResources` already fetches the character row)
- `discord/save_handler.go:281` — `buildSaveFeatureEffects(char refdata.Character)`
  (the caller `Handle` already fetched it via `GetCharacterByCampaignAndDiscord`)

→ Zero extra DB fetches needed. No combat-hot-path round-trip risk.

### Error handling
Inventory / attunement JSON unmarshal errors degrade silently to a nil slice —
mirrors the existing `Classes`/`Features` pattern in the same three funcs and
matches med-33 comment "better to drop a feature bonus than to fail the whole
roll." `ParsePassiveEffects` already swallows per-item parse errors inside
`CollectItemFeatures`. We do NOT want a corrupted single magic_properties row
to nuke an entire `/attack`.

### Will F-14 `modify_speed` reach the speed path?
Yes. `convertPassiveEffect` for `modify_speed` emits
`{Type: EffectModifySpeed, Trigger: TriggerOnTurnStart}`.
`turnStartSpeedBonus` calls `ProcessEffects(defs, TriggerOnTurnStart, ctx)`
and returns `result.SpeedModifier`, which `ResolveTurnResources` adds to
`speedFt` before exhaustion/condition halving. Verified in
`combat/turnresources.go:240`. Once `defs` includes the magic-item
FeatureDefinition, the modifier accumulates.

### Helper
Add a single private helper inside each of the three files (or a shared
exported one) that:
1. Parses `char.Inventory` → `[]character.InventoryItem` via
   `character.ParseInventoryItems`.
2. Parses `char.AttunementSlots` → `[]character.AttunementSlot` via
   `character.ParseAttunementSlots`.
3. Returns `magicitem.CollectItemFeatures(items, attunement)`.
4. Degrades on JSON error to nil.

To avoid a new package-level dep cycle, the helper lives at each call site
(attack.go, turnresources.go, save_handler.go). All three already import the
needed packages or can without cycles:
- `internal/combat` already imports `internal/character` (verified via
  `feature_integration.go` indirectly through `refdata`). Check needed.
- `internal/magicitem` imports `internal/combat`. So `combat` cannot import
  `magicitem` (would cycle).
  → For the combat package, inline the conversion (copy-paste of
  `magicitem.CollectItemFeatures` body is overkill; instead, the helper in
  combat will just call `magicitem.ItemFeatures` per item — except that also
  imports combat. So the helper in `combat` must parse the items and build
  the `FeatureDefinition`s directly using the existing `Effect` / `EffectType`
  symbols already available in the combat package.
  → Cleanest: extract the inventory→FeatureDefinition logic into a
  combat-internal helper. But this duplicates `magicitem.CollectItemFeatures`.
- Alternative: leave the helper in `magicitem` and have the call sites that
  CAN import `magicitem` (the discord save_handler) call it directly; for
  the two combat-internal call sites, inline a thin twin helper in combat
  itself. Decision: inline a `collectMagicItemFeatures` helper in
  `internal/combat/feature_integration.go` that mirrors
  `magicitem.CollectItemFeatures`. The duplication is small (≈30 lines),
  unavoidable given the import direction, and unit-tested by the existing
  `magicitem/integration_test.go` plus new tests below.

### Mirrored helper code-smell judgment

(a) **Import cycle is real.** `internal/magicitem/effects.go:8` imports
`internal/combat`. `internal/magicitem/effects_test.go:7` and
`integration_test.go:11` also import combat. No combat→magicitem import
exists. Reversing the direction would cycle. Confirmed by grep.

(b) **Mirror is faithful.** Reading `feature_integration.go:391-522`
side-by-side with `magicitem/effects.go:14-160`:
- `collectMagicItemFeatures` ≡ `magicitem.CollectItemFeatures` (same
  attunement set construction, same `IsMagic && Equipped` gate, same
  `RequiresAttunement` filter, same MagicProperties-first/MagicBonus-
  fallback ordering, same `Source:"magic_item"` tag).
- `magicBonusEffects` ≡ `magicitem.ItemFeatures` weapon/armor switch
  (same Effect types, same Triggers, same Modifier wiring).
- `parseMagicItemPassiveEffects` + `convertMagicItemPassiveEntry` ≡
  `magicitem.ParsePassiveEffects` + `convertPassiveEffect`. All eight
  cases match (modify_attack, modify_damage, modify_ac,
  modify_saving_throw, resistance, bonus_damage, modify_speed) with
  identical EffectType / Trigger / payload mapping.

(c) **Lockstep is documented.** Each of the four combat-side helpers
carries a "Twin of magicitem.X" or "must stay in lockstep with the
magicitem package" comment. The new combat tests cover the wiring; the
existing `magicitem/integration_test.go` covers the canonical
ProcessEffects path. Drift between the twins would surface as a combat
test failing while a magicitem test still passes (or vice versa).

**Judgment:** The duplication is ~50 lines, all in one file, well-named,
twin-commented, and exercised by both sides' tests. The plan
(`SR-006.md:72-83`) explicitly considered and rejected the third-package
extraction option (`internal/magiceffects`) because the magicitem
package already does the right thing and a third package would mostly
exist to satisfy graph topology. I agree with the trade-off — extracting
`magiceffects` would force `magicitem` into a thin wrapper and add a
package boundary purely for an internal import-graph constraint, which
violates "no over-engineering" from CLAUDE.md. The lockstep risk is
real but the surface area is small and CI catches drift via the two
parallel test files. **Approved as-is.** Recommend revisiting the third-
package extraction only if a third call site needs the same mirror.
