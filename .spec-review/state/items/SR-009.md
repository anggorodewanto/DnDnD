## Finding (verbatim from review)
`feature_uses` JSON shape is forked. `internal/combat/feature_integration.go:25` reads `map[string]int`; `internal/character/types.go:54` + `internal/rest/rest.go:170-178` read `{Current,Max,Recharge}`. Mutually unparseable — combat-written rage/ki rows never recharge on rest; dashboard-structured rows fail combat deductions.

## Acceptance criteria
- A single canonical Go type (`{Current, Max, Recharge}`) is used everywhere `feature_uses` is read or written.
- Combat write paths (rage, ki, etc.) write the structured form so `rest.go` can recharge them.
- Dashboard / rest write paths produce rows that combat read paths can decrement.
- Migration handles any existing legacy `map[string]int` rows in the DB (ask user before destructive ops).
- Round-trip test: combat writes a rage row → long-rest recharges it → combat reads it back at full uses.
- Tests pass; `make cover-check` green.

### Production write sites
- All twelve combat services above persist via `Service.SetFeaturePool` / `DeductFeaturePool` / `DeductFeatureUse` (one funnel in `feature_integration.go`).
- Rest: `internal/discord/rest_handler.go:417`, `cmd/dndnd/dashboard_apis.go:246` — both already marshal `map[string]character.FeatureUse`.
- DDB import: passes through `pqtype.NullRawMessage` opaquely (no seeding of feature_uses today; nothing to change).
- Portal builder: doesn't write feature_uses (confirmed by grep).

### Strategy
1. Change `ParseFeatureUses` signature from `(refdata.Character, string) -> (map[string]int, int, error)` to `(refdata.Character, string) -> (map[string]character.FeatureUse, int, error)`. Returned `int` is `fu.Current` for that key.
2. Change `SetFeaturePool` / `DeductFeaturePool` / `DeductFeatureUse` to take `map[string]character.FeatureUse`. They preserve `Max` / `Recharge` and only mutate `Current`. (If a key is missing — e.g. fresh seed without an entry — fall back to creating a `FeatureUse{Current: value}` entry; `Max`/`Recharge` will be filled when the character is seeded by the dashboard / portal / DDB import paths. This preserves backwards-compat with combat-only flows in tests.)
3. Combat callers thread the new type through unchanged code paths (variable names already named `featureUses`).
4. Update tests that build literal `{"ki":3}` style JSON to use the structured shape.
5. No DB migration. No conversion logic. App is not live.
