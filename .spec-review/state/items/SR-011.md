## Finding (verbatim from review)
Reckless first-attack gate broken. `attack.go:870` checks `cmd.Turn.ActionUsed`, which `Service.Attack` never sets — only decrements `AttacksRemaining`. Reckless can be re-declared on every Extra Attack swing. No test covers the negative path.

## Acceptance criteria
- `Service.Attack` (or its caller) sets/uses a field that `attack.go:870` reads to detect "first attack of the turn" — likely a `FirstAttackDone` flag or reading `AttacksRemaining` against the character's max attacks.
- Test: barbarian declares Reckless on first attack of turn — succeeds; declares Reckless again on Extra Attack swing — rejected with the spec error.
- Tests pass; `make cover-check` green.

## Worker plan

### State today
- `refdata.Turn` (internal/refdata/models.go:561) has `ActionUsed`, `BonusActionUsed`, `AttacksRemaining` — no dedicated "first attack" flag.
- `Service.Attack` orchestrates an attack via `UseAttack(turn)` which only decrements `turn.AttacksRemaining`. `ActionUsed` is never flipped for the multi-attack path (`ActionUsed=true` would block a second Extra Attack swing entirely because `ValidateResource(ResourceAttack)` short-circuits on `ActionUsed`). So the current gate at attack.go:887 (`if cmd.Reckless && cmd.Turn.ActionUsed`) is dead — it reads the wrong field.
- At the start of a turn, `ResolveTurnResources` (turnresources.go:217) seeds `attacks_remaining` to `resolveAttacksPerAction(char)` for PCs (max-of multiclass `attacks_per_action`) and 1 for NPCs. Reckless requires Barbarian class (HasBarbarianClass), so we always have a PC character record.

### Discriminator choice
- Compare `cmd.Turn.AttacksRemaining` against `resolveAttacksPerAction(char)` at the moment of the gate. They're equal iff no attack has been spent this turn → "first attack of the turn".
- Pros: zero new schema/field, zero new state writes, single boolean expression. Resets cleanly on turn start because `ResolveTurnResources` already re-seeds `attacks_remaining` per turn.
- The check sits BEFORE `ApplyLoadingLimit` mutates the working copy and BEFORE `UseAttack` decrements, so the un-decremented count is still in `cmd.Turn.AttacksRemaining`.
- No `FirstAttackDone` flag needed.

### Edge cases
- Multiclass barbarian: `resolveAttacksPerAction` already picks the max across classes, matching the value `ResolveTurnResources` used to seed the turn. Equality holds at swing 1.
- NPCs: blocked by HasBarbarianClass before reaching the new check, so no NPC-vs-PC max mismatch.
- Loading weapons: gate runs before `ApplyLoadingLimit`, so the cap doesn't poison the comparison.

Action Surge edge case (worker plan did not address):
- action_surge.go:57-58 resets ActionUsed=false AND AttacksRemaining=max. For a Fighter/Barbarian multiclass who Action-Surges, the gate would re-allow Reckless on the first surge swing because AttacksRemaining==max again. Spec line 682 says "first attack only" — most plausibly read as first attack of the TURN (matching the duration of the reckless self-disadvantage), not first of each action. The chosen discriminator is therefore slightly permissive in this niche multiclass + Action Surge combo. Mitigations: (a) only Fighter-multiclass Barbarians can Action-Surge, (b) the transient reckless condition (advantage to attackers) was already applied to the attacker on the first declaration and persists, so the practical mechanical leak is just an extra in-action advantage roll for the surge first swing. Recommend tracking as a follow-up nit (e.g., add a per-turn reckless_declared boolean, or have ActionSurge bump a counter that the gate consults). Not blocking SR-011 because spec is ambiguous and worker'''s discriminator matches the spec'''s likeliest reading at the level of the original failure.

Simpler or more fragile than FirstAttackDone bool?
- Simpler. Zero new schema, zero new Turn field, single equality check, automatic cross-turn reset via the existing ResolveTurnResources seeding. A FirstAttackDone bool would require: new column on turns, sqlc regen, persistence in UpdateTurnActionsParams, reset wiring in ResolveTurnResources AND ActionSurge AND any future feature that grants a fresh action. The derived check piggy-backs on the already-correct AttacksRemaining lifecycle. The one fragility (Action Surge re-arming the discriminator) would also need a FirstAttackDone-equivalent fix if the flag interpretation were tightened; that is a separate spec-interpretation question.
