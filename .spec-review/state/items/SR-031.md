## Finding (verbatim from review)
Explored cells in memory only. `mapRegeneratorAdapter.exploredCells` has no DB persistence — bot restart wipes every party's exploration history; incompatible with async-first design.

## Acceptance criteria
- `exploredCells` is persisted to the DB keyed by (campaign/party, map).
- On bot restart, exploration history is restored from the DB.
- Test: explore a few cells → simulate restart (or reload adapter) → previously-explored cells are still marked explored.
- Tests pass; `make cover-check` green.

## Worker plan

Persist the per-encounter explored-tile set as a JSONB column on the
existing `encounters` row instead of a separate table. The set is
rewritten as a single blob on each render — never per-tile mutated — so a
column outperforms a join table for this access pattern and keeps the
schema change minimal (one `ALTER TABLE`, no new keys / FKs).

Key invariants:
- The DB row is the source of truth. The adapter no longer holds an
  in-memory `map[uuid.UUID]map[int]bool` cache; it reads `enc.ExploredCells`
  on each render, applies it to the just-computed `FogOfWar`, unions the
  newly-Visible tiles, and writes the union back via
  `UpdateEncounterExploredCells`.
- A per-encounter `sync.Mutex` (lazily created, indexed in
  `exploredKM`) serializes the read-modify-write of one encounter's blob
  so two concurrent renders of the same encounter don't trample each
  other's union. Renders of *different* encounters proceed in parallel.
- Encoding is a sorted JSON array of int tile indexes (row*width+col,
  the same key the prior in-memory set used). Sorted output keeps the
  JSONB blob diffable in DB dumps; malformed payloads decode to an empty
  set (best-effort, so a corrupted row only loses dimming, never crashes
  the renderer).
- The legacy `applyExploredHistory` / `recordVisibleTiles` adapter
  methods are replaced by pure helpers (`applyExploredHistoryToFow`,
  `unionVisibleTilesInto`) that operate on a caller-supplied set. The
  existing unit test was updated to drive those helpers directly.

Migration approach: forward-only `ADD COLUMN ... DEFAULT '[]'`. No row
conversion required because the app is not live anywhere — existing
encounters simply start with an empty explored set and accrete from
the next render onward.

Test strategy (red/green TDD):
1. Pure-helper roundtrip test (`TestEncodeDecodeExploredCells_Roundtrip`)
   covers `decodeExploredCells` / `encodeExploredCells` including the
   sorted-output guarantee and the empty / malformed / negative-index
   defenses.
2. Acceptance integration test
   (`TestMapRegeneratorAdapter_ExploredCells_PersistAcrossRestart`):
   render with adapter A, then construct a fresh adapter B over the
   same fake-backed store; the previously-explored tiles must still
   appear in the persisted set after B renders. The fake's
   `UpdateEncounterExploredCells` mutates `f.encs[id]` so a later
   `GetEncounter` returns the updated row, exactly mirroring the live
   DB read-write loop.

Coverage: `discord_adapters.go` is already in `COVER_EXCLUDE` (it's a
thin Discord-side adapter, integration-tested in `cmd/dndnd`), so the
new helpers don't move coverage either way. `make cover-check` still
prints `OK: coverage thresholds met`.

Keying claim check: per-encounter scope is consistent with the design captured in `docs/phases.md:715` (exploration reuses the `encounters` table via a `mode` column; one map → one encounter). The originating finding only requires that a bot restart not wipe history; SR-031 acceptance criteria mention "(campaign/party, map)" but the spec model collapses that to per-encounter for the live system, so the narrower scope is defensible. Caveat: if a DM ever ends and re-creates an exploration encounter on the same map, the new encounter starts from `'[]'`. The worker plan acknowledges this as encounter-scoped by design. Recorded as a known limitation rather than a blocker — if a future spec change requires per-(campaign, map) persistence surviving encounter rotation, a follow-up SR can promote the column to a separate `(campaign_id, map_id) → explored_cells` table without changing the render loop.

Minor observations (non-blocking):
- `encodeExploredCells` (`discord_adapters.go:579-595`) hand-rolls insertion sort to "avoid pulling in sort". `sort.Ints` is in the std lib and already transitively imported elsewhere in the binary; the manual loop is fine but the rationale comment is slightly misleading. Style nit only.
- `renderInternal` holds the per-encounter lock across `renderer.RenderMap` (the PNG encode), not just the R-M-W of the JSON blob. For two concurrent renders of the *same* encounter this serializes the PNG step too; acceptable because (a) Discord debounces per channel upstream, and (b) the lock keeps the FoW computation paired with its eventual persist. Worth a brief inline comment in a future polish pass; not a defect.
- CLAUDE.md early-return: the new helpers (`decodeExploredCells`, `applyExploredHistoryToFow`, `unionVisibleTilesInto`, `persistExploredCells`) all use guard-clause early returns. Compliant.
