## Finding (verbatim from review)
Encounter zones persist but never render and never deal damage. `MapData.ZoneOverlays` exists but no production caller populates it; `ParseTiledJSON(..., nil, nil)` everywhere. `CheckZoneTriggers` returns `Effect:"damage"` but no caller rolls/applies — Spirit Guardians, Wall of Fire, Moonbeam, Cloud of Daggers don't damage. `ZoneAffectedTilesFromShape` also has no `line` case → Wall of Fire becomes a single tile.

## Acceptance criteria
- `cast_handler.go`, `move_handler.go`, `attack_handler.go` populate `MapData.ZoneOverlays` from `ListZonesForEncounter` before render.
- A central caller rolls and applies damage from `CheckZoneTriggers` `Effect:"damage"` results through the existing damage pipeline.
- `ZoneAffectedTilesFromShape` switch includes the `line` case (width + length).
- Tests: Spirit Guardians damages an enemy entering the zone; Wall of Fire deals damage along a line; Moonbeam damages on start-of-turn-in-zone.
- Tests pass; `make cover-check` green.

## Worker plan

### Where overlays already render
`cmd/dndnd/discord_adapters.go:466-470` (mapRegeneratorAdapter.renderInternal) already
populates `MapData.ZoneOverlays` from `ListEncounterZonesByEncounterID` before
`RenderMap`. That is the *only* PNG-producing path today; `done_handler.go` and
`enemy_turn_notifier.go` both go through `MapRegenerator.RegenerateMap`. So
the literal SUMMARY claim ("ParseTiledJSON(..., nil, nil) everywhere") is
*not* in the rendering path — the four `ParseTiledJSON(... nil, nil)` calls
inside cast/move/attack handlers are for non-render uses (wall lookup,
terrain/pathfinding) where overlays don't matter. Conclusion: the
"populate before render" criterion is already satisfied. Leaving those four
calls alone (no overlays needed for wall-fetching) is correct minimal change.

### The real gap: damage application
`CheckZoneTriggers` returns `Effect:"damage"` results, but the only consumer
is `PostZoneTriggerResultsToCombatLog` (done_handler.go:422) which posts a
"DM: roll damage" prompt to #combat-log — no automatic damage is rolled.
Same gap on the enter path in `Service.UpdateCombatantPositionWithTriggers`
(service.go:811) — results are returned but never applied.

### Damage applier design
Add `Service.ApplyZoneDamage(ctx, target, results)`:
- For each `ZoneTriggerResult` with `Effect=="damage"`, look up the spell via
  `store.GetSpell(SpellIDFromName(SourceSpell))`. SpellID convention is
  kebab-case slug of the spell name (e.g. "spirit-guardians", "wall-of-fire",
  "moonbeam"). Skip silently when spell row absent (test/seed gaps).
- Parse damage via `ParseSpellDamage(spell.Damage.RawMessage)`.
- Roll damage via `s.roller.Roll(dice)` (Service already has a roller; if not,
  inject minimal). Then funnel through `s.ApplyDamage(ApplyDamageInput{...})`
  reusing the existing damage pipeline (R/I/V, temp HP, exhaustion, etc.).
- Returns per-target outcome slice for log emission (best-effort log; do not
  fail the move/turn on missing data).
- Wire from two seams:
  1. `UpdateCombatantPositionWithTriggers` after `CheckZoneTriggers` —
     applies enter-trigger damage to the moving combatant.
  2. `createActiveTurn` after start-of-turn `CheckZoneTriggers` — applies
     start-of-turn damage to the combatant whose turn just started.

### Line shape
Add `case "line"` to `ZoneAffectedTilesFromShape`. The stored zone has only
`OriginCol/OriginRow`, `length_ft`, `width_ft` — no target direction stored
(zone_definitions does not capture caster direction). For the minimal patch
that covers the spec's Wall of Fire intent, render the line as
length×width axis-aligned east-from-origin (the same convention the
overlay/coverage code already implicitly uses). Tiles within the rectangle
`[origin..origin+lengthSquares-1] × [origin..origin+widthSquares-1]` form
the set. (Future: if a `direction` column is added to encounter_zones,
swap this for `LineAffectedTiles`.)

### Line geometry limitation (acknowledged, not a blocker)
- Stored zone has no direction column; the implementation renders any line
  east-from-origin. `Wall of Fire` cast at a non-axis-aligned target will be
  wrong. zone.go:333-340 documents this explicitly and references the future
  `LineAffectedTiles` swap. Minimal correct change for SR-014 scope; a
  follow-up SR can add a `direction` column to `encounter_zones` and switch
  to `combat.LineAffectedTiles`. Width default is the stored `width_ft`
  value floor-divided by 5; Wall of Fire seed JSON does not set `width_ft`,
  so it falls through to the `widthSquares < 1` clamp → width 1 tile.
  Acceptable for SR-014 (matches PHB 1ft thickness rounded up to a single
  5ft square).

### SpellIDFromName robustness
- zone.go:365-370 — trim, lowercase, space → '-'. Handles the four spec
  spells correctly (verified by `TestSpellIDFromName`). Will mis-slug
  apostrophes, slashes, or numerals (e.g. "Melf's Acid Arrow" →
  "melf's-acid-arrow" not "melfs-acid-arrow"). None of the current zone-
  emitting spells (Fog Cloud, Spirit Guardians, Wall of Fire, Darkness,
  Cloud of Daggers, Moonbeam, Silence, Stinking Cloud) trip that edge case,
  but a future spell with an apostrophe would silently no-op via
  `sql.ErrNoRows`. Acceptable for current scope — call this out in a future
  SR if more zone spells are added.
