## Finding (verbatim from review)
Exhaustion is read-only. Speed/HP/disadvantage/death effects wired, but nothing ever increments or decrements `exhaustion_level`. No `/exhaustion`, no dashboard mutation, no long-rest decrement, no forced-march hook.

## Acceptance criteria
- A mutation surface exists to increment/decrement exhaustion (slash command `/exhaustion` or DM dashboard mutation, per spec).
- Long rest decrements exhaustion by 1 (spec rule).
- Forced-march hook (per spec) can increment exhaustion.
- Tests cover increment, decrement, long-rest auto-decrement.
- Tests pass; `make cover-check` green.

## Worker plan
- Surface: DM dashboard mutation per spec line 1365 ("applied by DM from
  dashboard"). New endpoint:
  `POST /api/combat/{encounterID}/override/combatant/{combatantID}/exhaustion`
  body: `{"exhaustion_level": <0-6>, "delta": <int>, "reason": "..."}`.
  `delta` increments (or decrements via negative) when non-zero; absolute
  `exhaustion_level` is used only when `delta == 0` (DM-set absolute).
  Clamped to [0, 6]. Writes through `store.UpdateCombatantConditions`
  (existing query already takes `exhaustion_level`). Logs `dm_override`
  with before/after snapshot extended with `exhaustion_level`. Posts a
  combat-log correction.
- Forced-march hook: same endpoint — DM applies +N after a failed
  `/check constitution` per spec line 2740. No new code path needed.
- Long-rest decrement: add a pure helper
  `rest.LongRestExhaustionLevel(current int) int` returning
  `max(0, current-1)`. Wire it into `Service.LongRest` via a new
  `ExhaustionLevel` input + `ExhaustionLevelAfter` output so it always
  fires on long rest (no `/rest short` change). The result is exposed
  but persistence onto the combatant is deferred to SR-042 (which will
  resolve "find this character's active-encounter combatant and write
  exhaustion_level"); the helper + LongRest plumbing it adds here is
  what SR-042 will reuse. Add a one-line `> Exhaustion: level N` line
  to `FormatLongRestResult` when the value decreased.
- Tests:
  1. `combat`: `TestOverrideCombatantExhaustion_Set` — body sets level 3,
     store sees ExhaustionLevel=3, combat-log gets "exhaustion".
  2. `combat`: `TestOverrideCombatantExhaustion_Increment` — delta=+1 on
     a combatant with current=2 writes 3.
  3. `combat`: `TestOverrideCombatantExhaustion_DecrementFloor` — delta=-3
     on current=1 clamps to 0.
  4. `combat`: `TestOverrideCombatantExhaustion_ClampToMax` — delta=+5 on
     current=4 clamps to 6.
  5. `combat`: `TestOverrideCombatantExhaustion_InvalidBody` — 400.
  6. `combat`: `TestOverrideCombatantExhaustion_NoActiveTurn` — 404.
  7. `combat`: `TestOverrideCombatantExhaustion_GetCombatantError` — 500.
  8. `combat`: `TestOverrideCombatantExhaustion_UpdateError` — 500.
  9. `rest`: `TestLongRestExhaustionLevel_*` — pure helper: 0→0, 1→0,
     3→2, 6→5.
  10. `rest`: `TestLongRest_ExhaustionDecrements` — LongRest input
     ExhaustionLevel=3 produces result.ExhaustionLevelAfter=2.
  11. `rest`: `TestLongRest_ExhaustionFloorAtZero` — Input 0 → output 0.
  12. `rest`: `TestFormatLongRestResult_ExhaustionLine` — format renders
      the line only when level decreased.
- Out of scope: persisting LongRest's decrement onto the combatant row
  (SR-042). Heal-from-0 prone removal (different finding). Help action,
  Spare-the-Dying, temp-HP expiry, etc.

### Blocker
- `OverrideCombatantExhaustion` is added to `DMDashboardHandler.RegisterRoutes` (`internal/combat/dm_dashboard_handler.go:57`), but production traffic enters via `mountCombatDashboardRoutes` (`cmd/dndnd/main.go:317-347`), which binds each method individually and omits `/exhaustion`. Tests pass because `newDMDashboardRouter` (`internal/combat/dm_dashboard_handler_test.go:24-30`) uses `RegisterRoutes` directly.
- Consequences: 404 in the running server; SR-001 wiring assertion `TestMountCombatRoutes_RegistersWorkspaceAndDMDashboard` (`cmd/dndnd/main_wiring_test.go:817-834`) and SR-001 dmAuthMw table `dmOnlyRouteTable` (`cmd/dndnd/auth_dm_routes_test.go:171-173`) both miss it, violating the SR-019 confirm item "endpoint MUST be behind dmAuthMw".
- Fix: register `Post(".../exhaustion", dm.OverrideCombatantExhaustion)` in `mountCombatDashboardRoutes`; append to both wiring tables.
