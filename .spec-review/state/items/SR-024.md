## Finding (verbatim from review)
Paladin Aura of Protection not a registered class feature. `BuildFeatureDefinitions` has no `aura_of_protection` mapping — real paladins get no aura via `/save`. Tests inject manually.

## Acceptance criteria
- `BuildFeatureDefinitions` (or the appropriate registry) maps `aura_of_protection` as a real Paladin feature with the spec'd aura radius and CHA-mod bonus to saves.
- Test: paladin and ally within aura radius → ally's save gains paladin's CHA mod; ally outside radius → no bonus.
- Test does NOT inject the feature manually — it's resolved by the registry from the paladin's class data.
- Tests pass; `make cover-check` green.

## Worker plan

### Mechanical-effect ID
Paladins seeded in `seed_classes.go` will carry an `aura_of_protection`
`mechanical_effect` on the paladin level-6 feature row (spec line 2587: L6
self+allies-within-10-ft, L18 radius expands to 30 ft). For tests we exercise
the helper directly with the spec'd class shape so the seed change is not
strictly required for SR-024 — but the helper recognises the literal effect
string so a future seed update Just Works.

### Registry helper
Add `AuraOfProtectionFeature(chaMod int) FeatureDefinition` in
`feature_integration.go` (twin of `RageFeature` / `SneakAttackFeature`):
- `EffectModifySave` on `TriggerOnSave`, `Modifier=max(chaMod, 1)` (Paladin gets
  CHA bonus to saves; the floor matches PHB "min +1" via the underlying CHA mod
  arithmetic — actually 5e RAW does NOT floor to +1; the test cases use CHA 16
  → +3 so any non-negative value works. Keep the helper simple: just `Modifier:
  chaMod`).
- Source `"paladin"`.

Add `ResolvePaladinAura(classes []CharacterClass, chaMod int, paladinLevel int)
(FeatureDefinition, bool)` that returns the feature when `paladinLevel >= 6`,
else `(_, false)`. `BuildFeatureDefinitions` recognises the `aura_of_protection`
mechanical-effect string and, when CHA mod is passed via a new variadic option,
builds the aura — but the simpler design used here is a standalone helper that
save_handler invokes directly with the resolved level + CHA, since
`BuildFeatureDefinitions` does not currently carry ability scores. The
mechanical_effect literal is still recognised so the seed wiring round-trips
cleanly (it maps to a zero-mod placeholder when CHA is unknown — overwritten by
the dedicated helper at the save_handler layer).

### Nearby-paladin discovery in save_handler
- New optional `SaveNearbyPaladinLookup` interface with
  `GetCharacter(ctx, id uuid.UUID) (refdata.Character, error)`. Production
  wiring uses `refdata.Queries`; tests inject a small mock. Nil disables the
  feature (degrades to "no aura for allies" — matches the SR-006 silent-skip
  convention).
- New helper `nearbyPaladinAuras(ctx, char, encounterID)` in save_handler:
  1. Fetch encounter combatants via existing `combatantLookup`.
  2. Locate the saver's combatant by `CharacterID`.
  3. For each *other* combatant with a `CharacterID`, fetch the character via
     `paladinLookup.GetCharacter`. Skip on any lookup error.
  4. Determine paladin level from `char.Classes`; skip if `<6`.
  5. Compute aura radius: 10 ft at L6, 30 ft at L18.
  6. Compute Chebyshev grid distance via `combat.GridDistanceFt` between the
     saver and the paladin (already used by /shove, /grapple, /standard
     actions).
  7. If `distFt <= radius`, append the paladin's `AuraOfProtectionFeature(chaMod)`
     to the FES.
- Paladin self-aura: when the saver's *own* classes contain Paladin L6+, the
  helper above also picks it up (distance to self = 0 ≤ radius) — no separate
  code path needed.
