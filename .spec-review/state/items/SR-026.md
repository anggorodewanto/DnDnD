## Finding (verbatim from review)
`dm_required` spells and high-level teleports never create dashboard queue rows. Only set a string flag; no `dm_queue_items` insert; DM never sees them.

## Acceptance criteria
- Casts marked `dm_required` insert a real `dm_queue_items` row referencing the cast.
- High-level teleport casts insert a real `dm_queue_items` row.
- Insert-then-Discord-post ordering (matches SR-002 fix) — failed insert never posts.
- Test: cast a `dm_required` spell → row appears in `dm_queue_items` and is visible to the dashboard query.
- Tests pass; `make cover-check` green.

## Worker plan
- Add optional `notifier dmqueue.Notifier` and `campaignProv CheckCampaignProvider` to `CastHandler` (with `SetNotifier` / `SetCampaignProvider` setters mirroring the SR-002 reaction/undo handler pattern). Constructor stays compatible.
- After every successful `combatService.Cast` / `combatService.CastAoE` call in `runSingleTargetCast` / `runAoECast`, post a `dm_queue_items` row through the shared Notifier when the spell needs DM resolution:
  - Single-target: `result.ResolutionMode == "dm_required"` (covers both seeded `dm_required` spells *and* `IsDMQueueTeleport` narrative teleports, which `Cast` already promotes to `dm_required` at spellcasting.go:646).
  - AoE: `spell.ResolutionMode == "dm_required"` (AoECastResult does not currently carry ResolutionMode; the spell row already has it so we read it directly from the dispatch frame).
- Use `KindNarrativeTeleport` when the spell carries `spell.Teleport.Valid` AND the parsed target is in `IsDMQueueTeleport`; otherwise use `KindFreeformAction` with a "casts X (DM-resolved)" Summary so the dashboard sees a generic DM-required cast row.
- Resolve CampaignID via the new `campaignProv` (SR-002-style), falling back to "" when unset (Post short-circuits as silent no-op like the legacy callers).
- Wire `SetNotifier` + `SetCampaignProvider` from `cmd/dndnd/discord_handlers.go` next to the existing `handlers.cast.Set...` setters, threading `deps.notifier` and `checkCampProv`.
- Re-use the same Notifier instance; do NOT build a new one. The Notifier itself already does insert-then-send (SR-002), so the failed-insert-no-Discord-message invariant rides on that contract.
- Red/green TDD: failing tests first
  - `TestCastHandler_DMRequired_SinglePostsToDMQueue` — seeded spell with `ResolutionMode: "dm_required"` (e.g. fake "charm-person") → Notifier.Post called exactly once after Cast returns, Event.PlayerName=caster, GuildID/CampaignID populated.
  - `TestCastHandler_HighLevelTeleport_PostsKindNarrativeTeleport` — spell has Teleport JSON with target=group → Notifier post fires with `Kind == KindNarrativeTeleport`.
  - `TestCastHandler_DMRequired_InsertFailureSkipsDiscordPost` — recording-notifier that returns insert-error; ensure no Discord message goes out (the Notifier's SR-002 guarantee; we just pin the cast-handler does NOT manually `Send` outside the Notifier).
  - `TestCastHandler_AutoresolveSpell_NoDMQueuePost` — fire-bolt cast with `ResolutionMode != "dm_required"` → Notifier.Post NOT called.
  - `TestCastHandler_DMRequired_AoEPostsToQueue` — AoE spell with dm_required → Notifier.Post called from AoE path.
