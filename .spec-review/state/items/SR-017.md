## Finding (verbatim from review)
`/cast spare-the-dying` is seeded as auto-resolve but never wired to `StabilizeTarget`.

## Acceptance criteria
- The `/cast spare-the-dying` auto-resolve handler calls `StabilizeTarget` (or equivalent) on the target.
- Test: casting spare-the-dying on a downed creature stabilizes them and stops further death-save rolls.
- Test: casting on a creature at 0 HP without death saves yet — the rule from spec is applied correctly.
- Tests pass; `make cover-check` green.

## Worker plan
- Root cause: `/cast spare-the-dying` flows through `combat.Service.Cast` like any
  other auto-resolve spell, but Cast has spell-id-keyed post-effect branches
  only for Invisibility / Greater Invisibility (step 17) and zone creation
  (step 19). There is no branch that calls `StabilizeTarget` for the
  spare-the-dying spell, so the spell deducts no slot (cantrip) and posts the
  generic ✨ cast log line but leaves the target's death-save state untouched.
- Minimal change: add a new helper `applySpareTheDyingFromCast` on `*Service`
  in `internal/combat/spellcasting.go` that, when the cast resolves the
  `spare-the-dying` spell ID and a target is supplied:
    1. Read the target's current `DeathSaves` JSON via `ParseDeathSaves`.
    2. Skip when the target is not dying (alive at 0 HP, not yet stable) — the
       spell requires "a living creature that has 0 hit points" per SRD; for
       any other state we no-op silently so the cast still consumes the action
       (matches the existing pattern for already-invisible targets etc.).
    3. Call `combat.StabilizeTarget(target.DisplayName, ds, "Spare the Dying")`
       which already returns Successes=3 / TokenStable / IsStable=true,
       handling both the "downed creature with no death saves yet" case (the
       spec rule — 3 successes wins without rolling) and the "downed creature
       partway through death saves" case (override prior failures, mark
       stable).
    4. Persist via `s.store.UpdateCombatantDeathSaves(...)` using
       `MarshalDeathSaves(outcome.DeathSaves)`.
    5. Surface the outcome to the caller through a new `CastResult` field
       (`StabilizeMessage string`) appended to `FormatCastLog` so the combat
       log shows the 🩹 stabilized line directly under the ✨ cast header.
- Wire the branch in `Service.Cast` just before the invisibility branch
  (step 17 region), gated on `spell.ID == SpareTheDyingSpellID`. Introduce a
  package-level constant `SpareTheDyingSpellID = "spare-the-dying"` so the
  check has no string-literal magic.
- TDD cycles (red → green → refactor) in
  `internal/combat/spellcasting_test.go` (new file `spare_the_dying_test.go`
  to keep the existing test file's scope intact):
    * Cycle 1 (red): cast spare-the-dying on a downed creature mid-death-saves
      (Successes:1, Failures:2) → assert `StabilizeMessage` non-empty, target
      death saves persisted as Successes=3 (stable), `IsStable` outcome
      reflected, no slot consumed (cantrip).
    * Cycle 2 (red): cast on a creature at 0 HP with zero death saves yet →
      same stable outcome, Successes=3.
    * Cycle 3 (negative): cast on an alive >0 HP target → silent no-op,
      `UpdateCombatantDeathSaves` not called, cast still succeeds.
- Run `go test ./...` then `make cover-check`. Update Files touched. Set
  status `needs_review`. No commits.
