## Finding (verbatim from review)
Rage damage resistance has no consumer. `ProcessorResult.Resistances` populated but `internal/combat/damage.go` doesn't read it. Bludgeon/pierce/slash resistance likely doesn't apply.

## Acceptance criteria
- `internal/combat/damage.go` reads `ProcessorResult.Resistances` and halves matching damage types.
- Test: raging barbarian takes 10 slashing damage → 5 damage applied; 10 fire damage → 10 damage applied (rage doesn't resist fire).
- Tests pass; `make cover-check` green.

## Worker plan

### Path chosen
Rage's `EffectGrantResistance` is already produced by `RageFeature(...)` and
collected into `ProcessorResult.Resistances` whenever `EffectContext.IsRaging`
is true (see `internal/combat/rage.go:64` + `internal/combat/effect.go:328`).
SR-006 already wires `BuildFeatureDefinitions(classes, feats, magicItems...)`
through `populateAttackFES`. We do **not** need to extend the FES integration
or feature-definition catalog for rage; we just need a consumer in
`damage.go` that runs the same processor against the **target's** features at
`TriggerOnTakeDamage` and merges the resulting `Resistances` into the slice
already passed to `ApplyDamageResistances`. This is strictly shorter than a
generic "feature-driven R/I/V profile" overhaul.

### Where to plumb it
`Service.resolveDamageProfile(ctx, target)` is the single seam used by
`Service.ApplyDamage` to fetch the target's R/I/V slices before
`ApplyDamageResistances` halves damage. It currently returns:
- NPCs: creature row's `DamageResistances / Immunities / Vulnerabilities`
- PCs:  empty triple

We add a PC-side branch that, when the combatant is a PC with a valid
`CharacterID`, fetches the character row, parses `Classes` + `Features`,
calls `BuildFeatureDefinitions(...)` (no magic-item collection needed for
this issue — magic-item resistances would also work but they already require
a separate test; minimal change == classes/features only for now). Then
`ProcessEffects(defs, TriggerOnTakeDamage, EffectContext{IsRaging: c.IsRaging})`
yields `result.Resistances`, which we append to the resistance slice
before returning.

This preserves the canonical R/V/I order in `ApplyDamageResistances`
(immunity → R+V cancel → R halves → V doubles → petrified → none).
FES-driven resistances feed the same slice that NPC R/V/I currently use, so
the precedence ordering is already correct without any rewrite.

Failure modes degrade silently to "no extra resistances" (matches the
`buildSaveFeatureEffects` / `populateAttackFES` "drop the bonus rather than
fail the roll" convention from SR-006):
- `GetCharacter` errors → fall through to empty FES resistances
- JSON unmarshal errors → fall through to empty FES resistances

## Reviewer notes
- All checks pass: raging+slashing→5 (halved), raging+fire→10 (full), non-raging+slashing→10 (full). Three new tests `TestApplyDamage_RagingBarbarian{HalvesSlashing,FireUnaffected}` + `TestApplyDamage_NonRagingBarbarianTakesFullSlashing` pin the contract.
- R/V/I precedence preserved: FES resistances flow into the same `[]string` slice consumed by `ApplyDamageResistances`, so immunity→R+V cancel→R halves→V doubles→petrified ordering is untouched (no parallel code path).
- NPC path untouched (damage.go:383-393 unchanged): NPCs still pull `creature.DamageResistances/Immunities/Vulnerabilities` from the creature row and never enter `collectFESResistances`. Good — NPCs don't rage.
- Best-effort error handling matches SR-006: missing CharacterID, GetCharacter failure, malformed Classes/Features JSON all degrade to nil (lines 405-422). Unmarshal errors swallowed with `_ =` per the documented "drop the bonus rather than fail the roll" convention.
- Early-return style respected throughout `collectFESResistances`.
- Tests green; `make cover-check` OK (combat 92.82%).
- **Performance note (minor, not blocking):** `resolveDamageProfile` now triggers one extra `GetCharacter` call per damage event for *every PC target*, even non-raging ones with no FES-eligible features. `target.CharacterID` was already on the combatant row (sqlc-generated, no separate lookup for that), but the character row itself is a new DB hit on the damage hot path. Tolerable today since damage events are not high-frequency, but worth a follow-up optimization (cache on Service, or skip when `!target.IsRaging && target.CharacterID.Valid` and PC has no resistance-granting class) if Phase 48+ adds more PC R/V/I sources. Suggest filing as a tracked perf-debt note rather than blocking SR-021.
- One nit (also non-blocking): `collectFESResistances` swallows `json.Unmarshal` errors silently. SR-022+ should consider a debug-log line so malformed character JSON is at least observable.
