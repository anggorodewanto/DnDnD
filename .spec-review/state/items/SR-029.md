## Finding (verbatim from review)
Phase 21c lighting + elevation are write-only on the server. Svelte editor persists 5 Tiled layers; `internal/gamemap/renderer/parse.go` reads only `terrain` + `walls`. Lighting/elevation tiles stored but never consumed → magical-darkness FoW demotion only fires for runtime zones, not for map-painted lighting. (Compounded by Batch 10 obscurement gap.)

## Acceptance criteria
- `renderer/parse.go` parses the `lighting`, `elevation`, and `spawn_zones` Tiled layers (in addition to `terrain` and `walls`).
- Lighting tiles feed `LightSources`/`MagicalDarknessTiles` so the FoW pipeline (SR-008) sees them.
- Elevation tiles feed the cover/elevation rules per spec.
- Test: a map with a magical-darkness tile painted in the lighting layer demotes FoW correctly when parsed.
- Tests pass; `make cover-check` green.

## Worker plan

### Property schema (minimal, additive)

The Svelte editor writes 5 layers but encodes lighting GIDs directly from `LIGHTING_TYPES.gid` (small integers 1–5) rather than via the lighting tileset's `firstgid=7` offset (a separate drift covered by SR-030). To avoid touching SR-030 here, the parser supports BOTH paths:

1. **Preferred (Tiled-correct)**: lighting tileset entries carry `type` strings (`dim_light`, `darkness`, `magical_darkness`, `fog`, `light_obscurement`). GIDs in the lighting layer are matched against `firstgid + tile.id`.
2. **Fallback (matches today's editor output)**: if a layer GID does not resolve via tileset metadata, the parser falls back to the canonical `LIGHTING_TYPES` mapping (1=dim_light, 2=darkness, 3=magical_darkness, 4=fog, 5=light_obscurement). 0 = no override.

Light source radius: `LightSources` are NOT yet painted by the Svelte editor (the spec calls them out as "torches/Light cantrip" runtime sources, not map-baked). Per SR-029 scope ("static environmental lighting baked into the map"), `magical_darkness`/`fog`/`darkness`/`dim_light` are zones — they feed `MagicalDarknessTiles` and (future) generic obscurement tiles, not point `LightSources`. **Implementation parses magical_darkness explicitly today; the other lighting kinds are not consumed yet (SR-008 + Batch 10 obscurement gap) — the parse code records them as a no-op pass so it is forward-compatible.** This matches SR-029 acceptance criteria, which only require the magical-darkness FoW path to fire.

Elevation: the lighting/elevation layer data is row-major `[]int`. A new `MapData.ElevationByTile []int` (len = Width*Height, feet) carries the values. Map-baked elevation only affects token altitude defaults (per spec lines 321-336); the renderer keeps using `Combatant.AltitudeFt` for now (combatant state already supersedes ground elevation). **This SR only wires the parsed grid into `MapData`; downstream consumption (default altitude, cover from cliffs) is out of scope.**

Spawn zones: `internal/exploration/spawn.go` already parses these for encounter placement. The renderer's `MapData` gets a new lightweight `SpawnZones []SpawnZoneRect` field with pixel-to-tile-converted coordinates and `ZoneType` ("player"|"enemy"), populated by the renderer parser so future renderer features (DM-view zone overlays) can use them without redoing JSON parsing. Existing exploration code is unchanged.
