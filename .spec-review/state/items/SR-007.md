## Finding (verbatim from review)
Character cards never auto-update outside combat (Phase 17 partial). `OnCharacterUpdated` is fired only from `internal/combat/service.go`. None of `/equip`, `/use`, `/give`, `/loot`, `/attune`, `/rest`, `/prepare`, level-up, or DM inventory API trigger it — directly contradicts spec line 216. Compounded by `/equip` reading/writing different columns than the card (item 4).

## Acceptance criteria
- Every non-combat mutator publishes `OnCharacterUpdated` (or the equivalent character-changed event): `/equip`, `/use`, `/give`, `/loot`, `/attune`, `/rest`, `/prepare`, level-up flow, DM inventory API.
- A subscriber test asserts each handler emits exactly one event with the correct `CharacterID` on success.
- Spec line 216 contract is met: any mutation that changes a card-displayed field triggers a card refresh.
- Tests pass; `make cover-check` green.

## Worker plan

### Mechanism analysis

The existing seam is `combat.CardUpdater` (interface in `internal/combat/service.go:239`) with one method:
```go
OnCharacterUpdated(ctx context.Context, characterID uuid.UUID) error
```

`*charactercard.Service.OnCharacterUpdated` satisfies it. Production wires it via
`combatSvc.SetCardUpdater(cardSvc)` (cmd/dndnd/main.go:943). The fire-and-forget
contract is: silent on error, no-op on nil, no-op when no card message exists.

The interface lives in `internal/combat`. We can either:
(a) import `internal/combat.CardUpdater` from each handler/service package, or
(b) define a tiny, identical local interface in each package (same pattern combat
   uses for `DMNotifier` / `CardUpdater` itself to avoid import-cycle hazards).

Option (b) is what combat does. Several leaf packages (`levelup`, `rest`,
`inventory`) already follow this "minimal local interface + Setter" pattern
for `EncounterPublisher`. We will mirror that pattern: each package adds its
own minimal `CardUpdater` interface and a `SetCardUpdater(u CardUpdater)`
setter. No signature change to `OnCharacterUpdated`.

### Why route /equip through combat.Equip (not the handler)

After SR-004, production `/equip` goes through `combat.Service.Equip`, which is
the layer that actually mutates the character row (`UpdateCharacterEquipment`).
Firing the card update from `combat.Service.Equip` (one seam) is preferred to
firing from the handler (the handler doesn't have the post-Equip character ID
readily without re-reading the result). `combat.Service` already has
`notifyCardUpdateByCharacterID` defined exactly for this case but it's never
called. We will call it from `Equip` (after the final UpdateCharacterEquipment).

The handler's legacy `handleViaInventoryOnly` fallback also writes the inventory
JSONB; the card formatter reads `EquippedMainHand` (the column), so the
fallback's "Equipped:" line stays stale even with a card refresh. Still, the
spec contract is "any mutation that changes a card-displayed field triggers a
card refresh" — inventory items, attunement, and identification can change card
display indirectly through magic-item bonuses, so we fire from the fallback too.

### Why /interact is NOT in scope

`combat.Service.Interact` only mutates `Turn` and `PendingAction` rows — no
character-displayed field changes. The spec listing for SR-007 excludes
`/interact`. No fire site there.
