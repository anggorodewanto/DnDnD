## Finding (verbatim from review)
Narrative teleport poster framework-only — zero runtime callers. `PostNarrativeTeleport` + `KindNarrativeTeleport` fully tested but `/cast` never invokes them.

## Acceptance criteria
- `/cast` invokes `PostNarrativeTeleport` for spells/effects whose resolution is narrative teleport per spec.
- A `dm_queue_items` row of `KindNarrativeTeleport` is inserted, and the DM sees the prompt.
- Test: cast a teleport spell with narrative resolution → poster fires; DB row exists.
- Tests pass; `make cover-check` green.

## Worker plan

**No-op consolidation — already covered by SR-026.**

SR-026 (committed) wired the dm-queue post into `/cast` for `dm_required` casts via `CastHandler.maybePostCastDMQueue` (`internal/discord/cast_handler.go:782-797`). For narrative teleports (`result.Teleport != nil && result.Teleport.DMQueueRouted`), `castDMQueueKind` returns `dmqueue.KindNarrativeTeleport` (`cast_handler.go:826-831`) and `castDMQueueSummary` produces the byte-identical `"casts %s (narrative resolution)"` string (`cast_handler.go:838-843`) that `DefaultNotifier.PostNarrativeTeleport` emits (`internal/dmqueue/notifier.go:235-243`).

**Verification of the equivalence claim:**

1. `Notifier.Post` is the interface entry point; `PostNarrativeTeleport` is a method only on `*DefaultNotifier`. It is not on the `Notifier` interface, so the CastHandler cannot literally call `PostNarrativeTeleport` through its `dmqueue.Notifier` field without either extending the interface or doing a type assertion.
2. `PostNarrativeTeleport` does exactly one thing: construct an `Event{Kind: KindNarrativeTeleport, PlayerName: caster, Summary: fmt.Sprintf("casts %s (narrative resolution)", spellName), GuildID, CampaignID}` and delegate to `Post`.
3. SR-026's `maybePostCastDMQueue` constructs the same Event with the same Kind, PlayerName, Summary, GuildID, and CampaignID and calls `Notifier.Post`. The two paths therefore produce identical store rows and identical Discord message bodies.

**Plan:**

- No production code change. The acceptance criteria "a `dm_queue_items` row of `KindNarrativeTeleport` is inserted, and the DM sees the prompt" is satisfied today.
- Add one end-to-end TDD test that pins the runtime path **and** the equivalence with `PostNarrativeTeleport`, so if anyone changes `castDMQueueSummary` or the helper independently, CI fails before the drift reaches production.
- Test casts the seeded `teleport` spell with `DMQueueRouted=true`, runs the *real* `dmqueue.DefaultNotifier` (memory store + local fake sender), asserts a single pending row of kind `narrative_teleport` exists, and then calls `PostNarrativeTeleport` separately and compares the rendered message body byte-for-byte.

Red phase confirmed: temporarily flipping `castDMQueueSummary`'s teleport branch to a different string makes the new test fail with a diff at the message-body assertion (verified during development).

## Files touched

- `internal/discord/cast_handler_test.go` — added `TestCastHandler_SR027_NarrativeTeleport_RowExistsAndMatchesHelper`, the SR-027 contract pin. The test wires the *real* `dmqueue.DefaultNotifier` (memory store + local `sr027FakeSender`) into the cast handler, casts a high-level Teleport, and asserts that (a) the store ends up with exactly one row whose `Event.Kind == KindNarrativeTeleport`, and (b) the rendered Discord message body equals what `DefaultNotifier.PostNarrativeTeleport` produces for the same caster + spell. Added small local helper `sr027FakeSender` plus `stripResolveSuffixSR027` (mirrors the unexported `dmqueue.stripResolveLink`) so the comparison ignores the per-call resolve-link UUID.

No production code was modified. SR-026's wiring already satisfies the SR-027 contract; the new test pins the equivalence so future drift between the generic `Notifier.Post` path and the dedicated `PostNarrativeTeleport` helper trips CI.
