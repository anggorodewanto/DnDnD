## Finding (verbatim from review)
`/interact` bypasses `combat.Interact`. `internal/discord/interact_handler.go:108` calls `combat.UseResource(turn, ResourceFreeInteract)` directly — second `/interact` rejected instead of falling back to the action; no `pending_actions` row; DM never sees it.

## Acceptance criteria
- `internal/discord/interact_handler.go` routes through `combat.Interact` (the service that handles free-interact → action fallback and creates a `pending_actions` row).
- Second `/interact` on the same turn falls back to the action and does not error; the DM sees a queue/pending entry.
- Test covers: first `/interact` consumes free interact; second `/interact` consumes the action and creates the pending row; third `/interact` is rejected because action is spent.
- Tests pass; `make cover-check` green.

## Worker plan

Goal: replace the direct `combat.UseResource(turn, ResourceFreeInteract)` call in
`interact_handler.go` with `combat.Interact`, following the SR-004 pattern of
keeping a legacy fallback so existing tests without a wired combat service keep
working. The service already implements free→action fallback, auto-resolve
detection, `pending_actions` insert, and the action-spent error.

Mirror SR-004's `equip_handler.go` shape, scaled down:

1. Add a narrow `InteractCombatService` interface to `interact_handler.go`
   exposing only `Interact(ctx, combat.InteractCommand) (combat.InteractResult, error)`.
   `*combat.Service` already satisfies it.
2. Add `SetCombatService` setter that stores the service on `InteractHandler`.
   When nil, the handler keeps the current direct-UseResource path (legacy
   fallback). When set, route through `combat.Interact` instead.
3. In the combat-service branch:
   - Reuse the existing `encounterProvider.GetCombatant(turn.CombatantID)`
     lookup (already present below the UseResource call) and pass the
     resulting `Combatant` + `Turn` + description into `combat.InteractCommand`.
   - On success, build a single log line from `result.CombatLog` (covers both
     auto-resolved and DM-queue variants) and post to `#combat-log` via the
     existing helper.
   - Translate the service's "Free interaction already used and action is
     spent" error into the existing "Cannot interact: …" ephemeral.
4. In `cmd/dndnd/discord_handlers.go:516` (wiring), call the new
   `SetCombatService(deps.combatService)` setter so production picks up the
   new path. Guard with `deps.combatService != nil` (matches SR-004).
5. Failing tests first, in `internal/discord/interact_handler_test.go`:
   - (a) First `/interact` with combat service wired sets `FreeInteractUsed`,
     does NOT set `ActionUsed`, auto-resolves "draw longsword".
   - (b) Second `/interact` (turn.FreeInteractUsed=true) consumes the
     action and creates a `pending_actions` row (mock returns one); response
     mentions DM queue.
   - (c) Third `/interact` (both flags true) is rejected with the
     action-spent message and no row inserted.
   - Existing legacy tests stay green because they never call
     `SetCombatService`.
6. Test plumbing: spy combat service implementing the new interface, plus a
   way to capture `CreatePendingAction` — easiest is to back it with a real
   `combat.Service` over a fake `Store`. But importing `combat.Service` adds
   complexity; the existing `interact_test.go` already covers all three
   service-side paths. So the handler test can use a small spy that returns
   the canned `combat.InteractResult` values and records the inputs.
7. Run `go test ./...`, then `make cover-check`.
