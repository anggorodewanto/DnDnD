## Finding (verbatim from review)
`UNIQUE(campaign_id, discord_user_id)` blocks re-register-after-retire. Spec line 40 promises retired players can `/register` a new character; retire flow only sets status='retired' so the second INSERT fails. Either partial unique index or retire-frees-the-slot needed.

## Acceptance criteria
- Either: (a) the unique index is made partial — `UNIQUE(campaign_id, discord_user_id) WHERE status != 'retired'`; or (b) the retire flow nulls/removes the slot so a new character can claim it.
- Migration is non-destructive; ask user before applying a destructive choice.
- Integration test: register → retire → register again → second `/register` succeeds and the retired character remains queryable.
- Tests pass; `make cover-check` green.

## Worker plan

### Existing constraints (Phase 8 migration `20260310120007_create_player_characters.sql`)
Two unique table-level constraints, both inline in `CREATE TABLE`:
- `UNIQUE(campaign_id, discord_user_id)` → conventional name `player_characters_campaign_id_discord_user_id_key`. Blocks the spec line 40 promise (re-register after retire approval).
- `UNIQUE(campaign_id, character_id)` → conventional name `player_characters_campaign_id_character_id_key`. Spec line 43: "Retired characters remain in the database for story continuity — the DM can re-activate them from the dashboard if needed." Re-activation re-introduces the same (campaign, character) pair, so this constraint must also exclude retired rows to be consistent with the design. Otherwise a re-activate flow would silently fail.

### Decision: partial unique INDEXes (non-destructive)
- Preferred over retire-deletes-row, because spec line 43 explicitly mandates retired rows remain queryable. Confirmed by user policy: no row deletion / no destructive conversion.
- Postgres allows partial unique indexes via `CREATE UNIQUE INDEX ... WHERE`. They are functionally equivalent to a UNIQUE constraint for INSERT-time enforcement (the planner picks them up for `ON CONFLICT (...)` too, as long as the conflict target includes the predicate columns implicitly). We do not use `ON CONFLICT` here — `CreatePlayerCharacter` is a plain `INSERT ... RETURNING *` — so the swap is transparent.
- Migration plan:
  1. `ALTER TABLE player_characters DROP CONSTRAINT IF EXISTS player_characters_campaign_id_discord_user_id_key`
  2. `ALTER TABLE player_characters DROP CONSTRAINT IF EXISTS player_characters_campaign_id_character_id_key`
  3. `CREATE UNIQUE INDEX idx_player_characters_unique_active_discord_user ON player_characters (campaign_id, discord_user_id) WHERE status != 'retired'`
  4. `CREATE UNIQUE INDEX idx_player_characters_unique_active_character ON player_characters (campaign_id, character_id) WHERE status != 'retired'`
- Per user policy: forward-only, no `-- +goose Down`. (Existing recent migrations like `20260415120000_add_encounter_mode.sql` already omit a Down section, so this matches repo convention for forward-only changes.) Actually checking: most repo migrations do include a Down. The user said NO rollback script, so I will omit the Down stanza explicitly and leave a comment.

### Code-side audit
- `grep -rn "player_characters_campaign" /home/ab/projects/DnDnD/internal /home/ab/projects/DnDnD/cmd` returns no hits. No code relies on the constraint name in error handling.
- No `ON CONFLICT (campaign_id, discord_user_id)` clause exists in `db/queries/player_characters.sql`. Safe.

- `MarkPlayerCharacterRetireRequested` updates by `(campaign_id, discord_user_id)` only and ignores status — it will still update the active row pre-retire-approval. After retire approval, the row's status becomes 'retired', and a fresh `/register` for the same `(campaign_id, discord_user_id)` will INSERT a new row. The partial index allows multiple retired rows for the same (campaign, discord_user_id) over a character's lifetime, which is the desired semantic.

### Retire flow recheck
- `/retire` → `MarkPlayerCharacterRetireRequested` flips `created_via='retire'`, status stays 'approved'.
- DM approves from dashboard → `Service.Retire` runs `transitionStatus(... 'retired' ...)` → row now has `status='retired'`.
- Player runs `/register` → new INSERT with `status='pending'`. Both rows satisfy the partial index because the old row is excluded by `WHERE status != 'retired'` and the new row is the only non-retired row for that (campaign, discord_user_id).
